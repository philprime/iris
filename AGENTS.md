# AGENTS.md

**Iris** (`github.com/philprime/iris`) is a Kubernetes controller providing a single public SMTP entrypoint for a cluster that routes and transforms inbound mail to in-cluster services. A replicated Postfix ingress terminates public SMTP. The controller compiles `Relay` CRs (`iris.philprime.dev/v1alpha1`) into Postfix routing maps and reconciles one transformer pod per relay that filters, transforms (canonical JSON plus optional Jsonnet), and fans out to HTTP or downstream SMTP destinations. The public overview is the root [`README.md`](README.md). Design and contributing docs live under [`docs/`](docs/) (architecture, kubernetes, relay, observability, conventions). Replaces a legacy Postfix relay previously in the `infra` repo.

## Ground rules

- The `Makefile` is the front door for dev tasks (build, test, generate, format, lint, deploy, and so on). Run `make help` to discover targets and prefer them over the underlying commands. For example, use `make format` rather than `dprint fmt` or `go fmt` directly. Reach for the raw command only when no target fits.
- Never use compound shell commands. Do not chain separable commands with `&&`, `;`, or newlines, and do not wrap a sequence in `bash -c`. Run each command as its own invocation. Compound commands break the agent permission model, which grants approval one command at a time. A genuine pipeline (`a | b`) is fine because it is a single data flow.
- Run `git` plainly from the working directory (the project repo). Only use `git -C <path>` when targeting a different repo (e.g. the `infra` repo).
- Draft commit messages and PR descriptions as markdown files under `./tmp/` (e.g. `./tmp/commit-message.md`, `./tmp/pr-body.md`), then pass them in with `git commit -F ./tmp/commit-message.md` or `gh pr create --body-file ./tmp/pr-body.md`. If a pre-commit hook reformats files and aborts the commit, you can re-run from the same file without redrafting.
- Parse, query, or transform JSON of any kind with `jq`, and YAML of any kind with `yq`. Never use `python3` (or any other language/script) to read or manipulate these files, and do not hand-roll parsing.
- Write prose without em-dashes (`—`) or semicolons. Use full sentences instead. Split a joined clause into two sentences, or use a comma, parentheses, or a connecting word. This applies to docs, code comments, commit messages, and PR descriptions. It does not apply to code, inline code spans, or a lone `—` used as an empty or "not applicable" placeholder in a table cell.
- Keep docs free of details that are implemented elsewhere and can drift. When a config file or code is the source of truth (for example the hooks in `.pre-commit-config.yaml`, the targets in the `Makefile`, or the fields on a CRD), describe the intent and point to that file rather than restating its contents. Mention that a step exists, do not duplicate the list.
- Use [Conventional Commits](https://www.conventionalcommits.org/) (`feat`, `fix`, `build`, `chore`, `ci`, `docs`, `style`, `refactor`, `perf`, `test`). Branch from `main` with a type prefix such as `feature/`, `fix/`, `refactor/`, `docs/`, `chore/`, or `test/`. Keep `main` deployable.
- Never add AI or co-author attribution anywhere. No `Co-authored-by` trailers, no "Generated with" lines, and no robot emoji in commits, PRs, code, or comments.
- Go logging and errors: use `slog` with the `...Context` variants when a `context.Context` is in scope, structured fields only, and never `fmt.Sprintf` inside a message. Wrap errors with `fmt.Errorf("context: %w", err)` using lowercase strings (ST1005). Log an error or return it, never both. Full detail lives in [`docs/conventions.md`](docs/conventions.md).
- Verify before declaring work done. Run the relevant `make` targets (`make build`, `make test`, `make analyze`) and confirm they pass before committing or claiming the change is complete.
- Do not use the built-in file memory system. Persist any durable instructions or learnings by keeping this `AGENTS.md` up to date instead.
